# Harness Loop 文档设计（完整版手册）

## 背景与目标

`harness-loop.md` 当前仅包含极简流程示意，无法支撑新手完成首次搭建，也无法指导维护者进行长期运营与演化。本设计将其升级为一份中文完整手册，面向三类读者：

- 新手：第一次在项目中搭建 Superpowers Harness
- 进阶用户：需要稳定维护日常任务闭环
- 维护者/高级用户：需要基于失败证据持续演化系统

文档目标：

1. 讲清 Harness 的三段生命周期：搭建、维护、自我演化
2. 每章兼顾“原则 + 最小可执行步骤”
3. 建立可验证的产物与验收标准，避免“看起来完成但不可用”
4. 将高频故障处理路径显式化，减少重复踩坑

## 范围与非范围

### 范围

- 重写 `harness-loop.md` 为完整手册
- 采用“生命周期 + 角色视角”结构
- 对“搭建”给细粒度步骤
- 对“维护”给中粒度流程与检查点
- 对“自我演化”给策略框架与节奏建议
- 引用现有文档资产（如 hook 配置检查表、设计文档等）

### 非范围

- 不在本次文档中实现新脚本、新自动化或新 skill 行为
- 不修改现有核心 skill 的行为定义
- 不新增第三方依赖

## 信息架构

`harness-loop.md` 采用以下结构：

1. 文档定位（读者、目标、使用方式）
2. 核心心智模型（搭建→维护→演化闭环）
3. 第一部分：搭建（细粒度）
4. 第二部分：维护（中粒度）
5. 第三部分：自我演化（策略粒度）
6. 附录（快速入口、FAQ、术语）

## 章节写作模板（统一）

每个主章节都使用同一模板，降低维护成本并提高可扫描性：

1. 这一章解决什么问题（Why）
2. 最小可执行步骤（How）
3. 产物与验收标准（DoD）
4. 常见故障与恢复路径（Troubleshooting）
5. 下一步入口（Next）

## 三大章节详细设计

### 第一部分：搭建（细粒度）

目标：让第一次接触 Harness 的用户在单次阅读后完成最小可运行环境。

包含内容：

- Step 0：环境体检（仓库状态、基础文件、执行权限、目录可用性）
- Step 1：建立索引入口（`CLAUDE.md` / `AGENTS.md` 与 docs 入口关系）
- Step 2：生成知识蓝图（`harness-knowledge-discovery` 的使用时机与结果判定）
- Step 3：固化规则（`project-standards-authoring` 与规则文件落位）
- Step 4：接入任务后反馈（`harness-feedback-loop` 与 hooks）
- Step 5：首次验收（可观测信号 + 最小验收清单）

验收标准示例：

- 能明确定位任务前/后应读或写的文档路径
- Hook 配置满足检查表关键项
- 至少有一条完整执行记录落入 memory/log

### 第二部分：维护（中粒度）

目标：让系统在日常任务中持续稳定运行，避免文档与行为脱节。

包含内容：

- 任务前：读取上下文与约束（规则、checklist、历史 pattern）
- 任务中：偏差检测（计划偏移、边界蔓延、证据不足）
- 任务后：证据沉淀（成功/失败模式、执行日志）
- 周期性健康检查：结构完整性、文档时效性、失败复发率

验收标准示例：

- 每次任务都有可追溯执行记录
- 复发问题能在 failure patterns 中找到对应条目
- 规则更新与问题来源可建立映射

### 第三部分：自我演化（策略粒度）

目标：从“能跑”升级到“会学习”，形成低成本迭代机制。

包含内容：

- 演化触发条件（何时需要演化，何时应保持稳定）
- 决策框架：
  - 应更新文档？
  - 应更新 checklists？
  - 应新增/修改 skill？
- 演化节奏建议（双周/月度回顾）
- 反模式（无证据改动、一次改动过大、把偶发现象误当系统问题）

验收标准示例：

- 每轮演化都有“触发证据 → 变更内容 → 验证结果”链路
- 无证据的策略改动被显式拦截

## 文风与可读性约束

- 全文中文
- 使用“先原则、后步骤”的混合导向写法
- 面向三类读者，明确标注章节适用对象
- 控制术语密度，术语首次出现给一句定义
- 每节提供可执行完成信号，减少模糊表述

## 风险与缓解

1. **风险：文档过长导致不易执行**
   - 缓解：附录提供“按任务场景”的快速入口
2. **风险：章节重复、维护成本高**
   - 缓解：统一章节模板，减少异构写法
3. **风险：策略建议缺少证据约束**
   - 缓解：演化章节强制要求“证据链”结构
