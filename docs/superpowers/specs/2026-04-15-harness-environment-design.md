# Harness Environment Design

## Overview

Build a document-driven Harness environment on top of Superpowers that improves Agent stability and autonomy in complex tasks, establishes a reusable framework for constructing Harness environments across different project types, and enables self-evolution through failure learning and memory persistence.

## Goals

| Goal | Description | Phase |
|---|---|---|
| A. Improve Agent stability | Agents retrieve project specs autonomously, avoid repeating errors, and stay on plan | Phase 1 |
| B. Establish reusable framework | Generate customized knowledge base blueprints for any project type (frontend, Java, Node, etc.) | Phase 1 |
| C. Enable self-evolution | Learn from failures, persist memory across sessions, iterate skill effectiveness | Phase 2-3 |

## Architecture

### Core Principle

**Skills define "how" (behavior), docs provide "what" (knowledge).**

- Superpowers skills provide generic Harness capabilities shared across all projects
- Project `docs/` contains project-specific knowledge that Agents read to constrain their behavior
- `CLAUDE.md` (or `AGENTS.md`) serves as the index entry point, auto-loaded by Cursor and Claude Code
- `agent.md` serves as the Harness index, referenced from `CLAUDE.md`/`AGENTS.md`

### Component Diagram

### Dual-Agent Architecture

The Harness environment enforces a strict separation between planning and execution:

**Orchestrator (Main Agent)**:
- Understands user requirements
- Creates design and implementation plans
- Dispatches tasks to Worker Agents
- Reviews Worker Agent output against specs and checklists
- Decides: proceed, rework, or deliver
- Holds global context, does not write implementation code

**Worker (Sub Agent)**:
- Receives a single, well-defined task
- Operates in an isolated context (no history from previous tasks)
- Reads relevant docs/ for constraints and knowledge
- Produces concrete output (code, docs, tests)
- Does not modify the plan or scope

This separation prevents context pollution, enables independent review, and limits deviation risk by keeping individual tasks to 2-5 minutes.

### Phase 1 vs Phase 2-3 Boundary

Phase 1 delivers the foundational layer (index, knowledge discovery, feedback collection). Phase 2 adds failure pattern analysis and learning mechanisms. Phase 3 adds cross-session memory persistence and skill effectiveness evaluation. Phase 1 must prove viability before Phase 2-3 work begins.

## Skills

### Skill 1: harness-knowledge-discovery

**Purpose**: Analyze a project's source code, directory structure, configuration files, and all available resources to generate a customized knowledge base blueprint — the list of docs/ files an Agent needs to work stably and autonomously on complex tasks for that specific project.

**Gap from existing `project-standards-authoring`**:

| `project-standards-authoring` | `harness-knowledge-discovery` |
|---|---|
| Extracts coding rules from existing code and docs | Analyzes project structure to infer what knowledge docs are missing |
| Outputs rules/ files | Outputs a complete knowledge base blueprint (rules + harness + memory + checklists) |
| Generic project | Project-type-aware (frontend, Java, Node, microservice, etc.) with differentiated templates |

**Workflow**:
1. Scan project structure (package.json, pom.xml, directory layout, configs, etc.)
2. Identify project type and architecture pattern
3. Infer knowledge gaps:
   - Tech stack constraints → need tech stack spec docs
   - Architecture patterns → need architecture convention docs
   - Common failure modes for this project type → pre-fill failure pattern library
   - Quality gates appropriate for this project type → generate customized checklists
4. Output knowledge base blueprint: a structured list of docs/ files to create, with rationale for each

**Collaboration with `project-standards-authoring`**:
- Run `harness-knowledge-discovery` first to get the blueprint
- Run `project-standards-authoring` to fill in specific rule content

### Skill 2: harness-feedback-loop

**Purpose**: Automatically collect execution evidence after task completion, compare expected vs actual results, and write failure/success patterns to `docs/memory/`.

**Workflow**:
1. Trigger after task completion (via hooks or manual invocation)
2. Compare planned output against actual output
3. Identify deviations, errors, or unexpected successes
4. Write structured entries to:
   - `docs/memory/failure-patterns.md` (if task failed or deviated)
   - `docs/memory/success-patterns.md` (if task succeeded with notable patterns)
   - `docs/memory/execution-log/YYYY-MM-DD.md` (always, for traceability)

**Pattern Entry Format**:

```markdown
## [Pattern Name]
- **Trigger Condition**: When...
- **Symptom**: ...
- **Root Cause**: ...
- **Resolution**: ...
- **First Recorded**: YYYY-MM-DD
- **Occurrence Count**: N
```

## agent.md Index Mechanism

`agent.md` is referenced from `CLAUDE.md`/`AGENTS.md` (which are auto-loaded at session start). It uses a hybrid indexing approach:

### Core Index Table

A quick-reference table mapping needs to docs:

```markdown
| When you need... | Read |
|---|---|
| Project coding standards | docs/rules/index.md |
| Handle test failures | docs/memory/failure-patterns.md |
| Start a new task | docs/checklists/pre-task.md |
| Review code | docs/checklists/code-review.md |
```

### Scenario Trigger Rules

Specific mappings from task context to required docs:

```markdown
- **New feature** → load docs/rules/*.md + docs/checklists/pre-task.md
- **Debugging** → load docs/memory/failure-patterns.md
- **Code review** → load docs/checklists/code-review.md + docs/rules/*.md
- **Task complete** → execute docs/checklists/post-task.md + write to docs/memory/execution-log/
```

## Platform Support

Phase 1 targets Cursor and Claude Code. Cursor auto-loads `AGENTS.md`, Claude Code auto-loads `CLAUDE.md`. Both are kept in sync (symlink or dual-write). The design avoids platform-specific code, keeping docs and skills harness-agnostic.

## Success Criteria for Phase 1

Phase 1 is validated when all three criteria are met:

| Criterion | Evidence | Measurement |
|---|---|---|
| **A. Functional** | Agent auto-retrieves specs from docs/ and follows them without explicit prompting | Observe session: Agent reads correct docs before acting |
| **B. Efficiency** | Sub-Agents deviate from plan fewer times, autonomous work sessions extend | Track via execution-log: count deviations per session |
| **C. Quality** | Code review catches fewer spec-related issues, recurring problems stop appearing | Compare review issue counts before/after Harness deployment |

## Scope

Phase 1 delivers:
1. `agent.md` index file with hybrid indexing (core table + scenario rules)
2. `docs/harness/` directory with project-specific architecture notes
3. `docs/memory/` directory structure with initial pattern files
4. `docs/checklists/` directory with pre-task, code-review, and post-task checklists
5. `harness-knowledge-discovery` skill
6. `harness-feedback-loop` skill
7. Hooks configuration for Cursor and Claude Code

Phase 1 does NOT deliver:
- Cross-session memory persistence beyond file-based storage (Phase 3)
- Vector search or embedding-based retrieval (out of scope, introduces dependencies)
- Automatic skill effectiveness evaluation (Phase 3)
- Modifications to existing Superpowers core skills (all new skills are additions)

## Skill Creation Process

All skills MUST be created following the `superpowers:writing-skills` TDD workflow:

1. **RED**: Run baseline scenario WITHOUT the skill, document exact agent behavior and rationalizations
2. **GREEN**: Write minimal skill addressing those specific rationalizations, verify compliance
3. **REFACTOR**: Find new rationalizations, plug loopholes, re-test until bulletproof

See `superpowers:writing-skills` for the complete methodology including pressure testing, rationalization tables, and red flags.

## Constraints

- `agent.md` MUST stay under 200 lines to remain scannable and token-efficient
- Zero external dependencies: all storage uses local Markdown/JSON files
- No modifications to Superpowers core skill behavior (only additions)
- All generated docs must follow the existing `docs/` conventions
- Skills must be testable via existing Superpowers testing infrastructure

## Migration Notes

### standards-context-retrieval

This skill is currently a required sub-skill of `project-standards-authoring`. Under the new design, `CLAUDE.md`/`AGENTS.md` auto-loading with a reference to `agent.md` eliminates the need for dynamic context retrieval as a separate skill.

**Migration plan**:
- Phase 1: `project-standards-authoring` continues to use `standards-context-retrieval` as-is (no breaking changes)
- Phase 2: Evaluate replacing `standards-context-retrieval` with direct `CLAUDE.md`/`AGENTS.md` + docs/ retrieval in `project-standards-authoring`
- If replaced: deprecate `standards-context-retrieval` and update all references in tests, plans, and validation scripts
