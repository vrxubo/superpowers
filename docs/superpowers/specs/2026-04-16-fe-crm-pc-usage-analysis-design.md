### 1.1 cursor_.md - 需求开发讨论

**使用的 Skill**：`brainstorming`、`TDD`

**人类使用模式**：
```
用户输入序列：
1. "/brainstorming 根据需求文档...使用lark-mcp阅读文档"
2. "重新使用lark-mcp阅读文档，我已经完成授权"
3. "继续"  (被重复输入多次)
4. "A" - 选择范围
5. "B" - 选择方案
6. "确认" (被重复输入多次)
7. "你可以重新读一下接口文档..."
8. "进入下一步"
9. "1" - 选择方案编号
10. "确认" (多次)
11. "进入下一步 writing-plans"
12. "1"
13. "/brainstorming 再次开始新需求"
14. "/brainstorming 基于当前项目情况帮我定制当前项目的单元测试方案"
15. "A" / "B" / "C" - 选择
16. "方案2"
17. "确认" (多次)
18. "进入下一步 writing-plans"
```

**观察到的行为模式**：

| 行为 | 频率 | 说明 |
|------|------|------|
| 使用 `/brainstorming` 触发 | 高 | 每次需求都用 |
| 快速确认（单字回复 A/B/确认）| 高 | 熟练使用单键确认 |
| 重复输入"确认"/"继续" | 中 | 可能等待时间过长或缺乏反馈 |
| 主动提供额外上下文 | 中 | 如"我已经完成授权" |
| 进入下一步技能 | 中 | brainstorming → writing-plans |

**效率问题识别**：
1. **等待反馈时间过长**：用户多次输入"继续"/"确认"，说明 Agent 响应后用户需要等待才能理解可以继续
2. **方案选择快速**：用户能快速在 A/B/C 选项中做选择，说明多选项设计有效
3. **TDD 流程复杂**：用户选择"方案2"而非推荐的"方案1"，可能是因为测试基础设施已经存在

## 五、Superpowers 优化建议

### 5.1 短期优化（立即可做）

#### 5.1.1 brainstorming Skill 优化

**问题**：用户多次输入"继续"/"确认"，响应后状态不清晰

**优化方案**：
```
在 Agent 响应末尾增加明确的行动指示：

现状：
"请确认以上设计是否 OK？"

优化后：
"✅ 设计方案已提交。你可以：
   1. 确认并进入下一步 → 输入 "确认"
   2. 修改某点 → 指出需要修改的内容
   3. 查看完整设计稿 → 输入 "设计稿"

当前等待你的回复：_______________
```

**问题**：5个段落逐一确认太冗长

**优化方案**：
```
支持批量确认：

"以上 5 个段落你是否全部确认？
   - 全部确认 → 输入 "全部确认"
   - 有修改 → 指出段落编号和修改内容
```

#### 5.1.2 TDD Skill 优化

**问题**：Mock 复杂性导致测试编写困难

**优化方案**：
```
在 RED 阶段前增加"可测试性检查"：

"在开始编写 RED 测试前，我先检查一下模块的可测试性...
模块 A：✅ 可直接测试
模块 B：⚠️ 导入链过深，建议先做以下调整：
   1. [具体建议]
   2. [具体建议]
   
是否继续？"
```

### 5.2 中期优化（需要规划）

#### 5.2.1 会话状态透明化

**建议**：在每次 Agent 响应中包含当前状态

```
当前状态：[brainstorming - 方案确认阶段]
进度：3/7 步骤已完成
上一步：选择了方案A（最小侵入）
下一步：将进入 writing-plans 生成实施计划

你的选项：
[1] 确认当前方案
[2] 重新选择方案
[3] 修改某个细节
```

#### 5.2.2 快捷命令增强

**观察**：用户习惯使用 `/brainstorming` 等快捷命令

**建议**：
1. 增加更多快捷命令覆盖常见场景
2. 支持 `//` 简写形式
3. 快捷命令应有清晰的完成信号

### 5.3 长期优化（战略层面）

#### 5.3.1 学习人类使用模式

**观察到的学习点**：
1. 人类善于一次性给出完整信息
2. 人类习惯快速确认而非逐项审查
3. 人类会主动补充上下文

**建议**：
- 调整 brainstorming 默认行为，支持"快速模式"
- 减少确认轮次，默认使用批量确认
- 增加"上下文记忆"，减少重复信息提供

#### 5.3.2 适应不同熟练度

**观察**：
- 新用户：可能需要更多确认
- 熟练用户：希望快速收敛

**建议**：
- 增加"熟练度检测"
- 熟练用户使用简化流程
- 新用户使用完整引导流程

## 六、产出文档

### 6.1 面向 Superpowers 维护者

**优化建议汇总**：

| Skill | 问题 | 优化建议 | 优先级 |
|-------|------|---------|--------|
| brainstorming | 响应后状态不清晰 | 增加明确的行动指示和进度显示 | P0 |
| brainstorming | 确认流程过长 | 支持批量确认 | P1 |
| TDD | Mock 复杂 | 增加可测试性预检 | P1 |
| project-standards-authoring | 重复文档 | 增加自动检测和清理 | P2 |

### 6.2 面向人类使用者

**最佳实践指南**：

#### 使用 brainstorming 的最佳方式

1. **准备阶段**
   - 整理好需求文档和接口文档链接
   - 确认需求范围和约束条件
   - 准备好几张关键截图（如有 UI 变更）

2. **执行阶段**
   - 使用 `@` 引用已有的设计文档
   - 一次性给出完整的需求清单
   - 在澄清问题时补充尽可能多的细节
   - 明确选择方案而非模糊回复

3. **确认阶段**
   - 逐段落确认时，可以一次性确认多项
   - 确认"进入下一步"后主动触发下一个 skill

#### 使用 TDD 的最佳方式

1. **前置检查**
   - 确认项目已有测试基础设施（vitest/jest）
   - 了解现有测试的 mock 模式

2. **RED 阶段**
   - 写出明确的预期行为
   - 避免过度设计测试

3. **GREEN 阶段**
   - 优先实现最简单的解决方案
   - 不要试图一次性通过所有测试

4. **REFACTOR 阶段**
   - 小步重构，每次只改一件事
   - 保持测试始终通过

#### 常见问题解决

| 问题 | 解决方案 |
|------|---------|
| Agent 响应后不知道该输入什么 | 查看 Agent 给出的选项，通常是 A/B/C 或数字 |
| 想要重新开始 | 输入 `/brainstorming` 开始新话题 |
| 发现之前的选择有问题 | 直接指出需要修改的内容 |
| 想跳过某些确认 | 输入"全部确认"或"跳过" |
